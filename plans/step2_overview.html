1. Step 2 개요

Step 2는 LLM 기반 multi-agent reasoning을 통한 구조화된 미세 보정 단계이다. Stage 1에서 탐색 공간이 축소되고 잔차가 구조적으로 정리된 시점부터 시작하며, 자유 생성 AI 후처리가 아니라 사전 정의된 제약·어휘·채택 규칙 안에서 운영된다.

운영 원칙

  • 변수 어휘 고정: Step 0에서 정의된 22개 변수 pool 전체가 Stage 2 작업 대상. 신규 변수 정의·생성 불가
  • Stage 1 status는 prior: active/freeze/미활성 모두 후보, 단 잔차 시그니처가 지목해야 freeze 변수가 재활성
  • diagnostic-driven routing: diagnostic agent가 잔차 시그니처를 해석 → orchestrator가 라운드별 specialist subset을 동적 할당
  • 값 수정 권한 분리: diagnostic·orchestrator·reality는 변수 값 수정 불가 / specialist만 후보안 제출
  • 감사 로그: 전 단계가 변수 identifier 어휘로 로깅되어 structural reproducibility 보장

2. Agent 토폴로지

DIAGNOSTIC 잔차 진단 ORCHESTRATOR 라우팅·채택 SPECIALIST 후보안 제출 REALITY 사후 검토

2.1 Diagnostic agents (2)

electricity-diagnostic-agentDIAG
전력 측 residual pattern을 진단하는 critic. 최종 case를 생성하지 않고 specialist 호출 근거를 제공.
입력: 실측 전력, score.json, monthly_comparison.csv, monthly_results.json, specialist 후보안 요약
출력: 전력 잔차 시그니처 + 정합 variable candidate list
gas-diagnostic-agentDIAG
가스 측 residual pattern을 진단하는 critic. 전력과 동일한 책임 구조.
입력: 실측 가스, score.json, monthly_comparison.csv, monthly_results.json, specialist 후보안 요약
출력: 가스 잔차 시그니처 + 정합 variable candidate list

두 diagnostic의 교차 시그니처(예: 여름 cooling 전력 bias + shoulder 가스 bias)는 orchestrator가 통합 해석한다.

2.2 Orchestrator agent (Phase 1 / Phase 2 두 시점)

orchestrator-agentORCH

Step 2 전체 흐름을 관리하는 최종 의사결정 agent. 변수 *값* 수정 권한 없음 — routing·할당·채택·freeze만 수행.

Phase 1 — 라운드 시작

  • diagnostic의 candidate list로부터 variable subset 분할
  • specialist 인스턴스 호출 + 동적 할당
  • routing prior(물리 직관) 참고하되 컨텍스트 따라 재해석 가능

Phase 2 — 라운드 종료

  • specialist 후보안 + reality 판정 + 전역 J 평가 종합
  • 채택 / hybrid / 기각 결정
  • per-variable 변동 < θ_freeze 기준 freeze 갱신

2.3 Specialist agents (6) — 카테고리별 후보안 제출

orchestrator가 라운드별 동적으로 호출. 같은 라운드에 여러 인스턴스가 서로 다른 variable subset을 담당하며 병렬 호출. 출력은 (변수 식별자, 신규 값, 물리 사유) tuple list.

envelope-air-exchange-agentSPEC
외피 및 공기교환 카테고리.
담당: infiltration 관련 변수 / 외피 U-value, SHGC, 침기율 등
internal-loads-agentSPEC
내부발열 및 전력 기저부하 카테고리.
담당: lighting / equipment / occupancy density multiplier
schedules-operation-agentSPEC
운영 스케줄 및 availability 패턴 카테고리.
담당: occupancy/lighting/equipment schedule scale·shift, HVAC availability
thermostat-setpoints-agentSPEC
thermostat 및 setpoint 제어 카테고리.
담당: heating/cooling setpoint offset, setback
hvac-plant-efficiency-agentSPEC
HVAC 및 plant efficiency 카테고리.
담당: boiler efficiency, COP, fan/pump 효율
dhw-agentSPEC
급탕 수요 및 DHW 제어 카테고리.
담당: DHW use multiplier, DHW 운영 가정

2.4 Reality agent (1) — 사후 검토

reality-agentCRITIC

specialist 후보안 사후 검토에만 호출됨 (라운드 시작 시점에는 호출되지 않음). pass / hold / reject + 사유 출력. 변수 값 수정 권한 없음.

  • Hard physics 위반 (ASHRAE 90.1/G14/62.1, 국토교통부 건축물 에너지절약설계기준 등 외부 표준): 자동 reject
  • Soft plausibility: 표준 범위 내 결합 모순(예: 외피 매우 우수 + 침기율 매우 높음) → hold/pass 의견

3. 라운드 흐름 (한 라운드 sequence)

잔차 진단 — diagnostic agents 병렬 호출
electricity-diagnostic + gas-diagnostic이 월별 잔차·변수별 J 기여도를 해석해 시그니처 + variable candidate list 생성. 22개 pool 전체에서 도출.
라우팅 — orchestrator phase 1
candidate list를 specialist 카테고리별 subset으로 분할 → specialist 인스턴스 호출 명령(할당 subset + 라운드 컨텍스트)을 발행.
후보안 제출 — specialist 병렬 실행
호출된 specialist들이 할당 subset 안에서 (변수 식별자, 신규 값, 물리 사유) tuple list를 반환. 할당 외 변수·freeze 변수 수정 불가 (schema validation).
사후 검토 — reality agent
각 후보안에 대해 hard physics binary reject + soft plausibility hold/pass 의견을 출력.
시뮬레이션·평가
후보안 적용 OSM으로 EnergyPlus 실행 → 월별 CVRMSE/NMBE → J 재계산.
채택·freeze — orchestrator phase 2
단일 우세 / 비충돌 hybrid / 기각 + per-variable 변동 < θ_freeze인 변수 freeze. hold + J 개선 미미는 다음 라운드 재고. 모든 결정은 변수 어휘로 감사 로그.

4. 채택 규칙 · 제약 · 종료 조건

4.1 채택 규칙 (orchestrator phase 2)

  • 단일 specialist 후보안 우세 → 그대로 채택
  • 복수 후보안이 비충돌·상보적인 경우에만 hybrid 허용 (서로 다른 subset)
  • reality reject → 자동 기각
  • reality hold + J 개선 미미 → 다음 라운드 재고
  • 임의 합성·자유 생성 채택 금지

4.2 제약 조건

  1. 도메인 제한: subset 외부·22개 어휘 외·Stage 2 freeze 변수 수정 불가
  2. Hard physics: 외부 표준 위반 자동 reject
  3. Soft plausibility: 결합 모순 hold/pass 의견
  4. 채택 규칙 제한: 사전 정의된 규칙만 허용
  5. 감사 로그: 전 단계 변수 identifier 어휘 로깅
